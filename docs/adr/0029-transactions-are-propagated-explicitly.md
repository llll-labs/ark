# ADR 0029: Transactions are propagated explicitly

Ark v1 follows the Directus service pattern for transaction propagation. The top-level Resource or Domain Service owns the operation transaction. Filters and required Actions use that transaction. Their Hook Context exposes a transaction-bound `services` factory, so `context.services.resource(name)` joins the current operation automatically. Domain Services composing several writes use `withArkResourceTransaction()`, whose database and service factory are explicitly scoped to the transaction. Ark does not provide a global ambient transaction.

This keeps the v1 service model close to Directus and makes transaction participation visible at the composition boundary. Callers coordinating Resources and Internal Tables must use the database and services supplied by that boundary consistently. Durable external effects still use the transactional outbox. Best-effort Actions run only after commit and therefore receive a normal database connection and a new service factory, not the completed operation transaction.

The tradeoff is deliberate: constructing an unrelated service outside the supplied context can still run outside the parent operation. A Resource Service therefore rejects an arbitrary `transaction` object; transaction-bound services must come from `withArkResourceTransaction()` or the Hook Context factory so Ark can preserve the operation's required-Action and post-commit queues. Tests and service APIs should make the composition boundary obvious.
